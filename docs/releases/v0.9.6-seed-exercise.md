# v0.9.6 `$50` Seed Exercise

This is the constrained live-funds validation plan for the first complete self-funding mechanism.

The goal is not to prove long-run profitability with a tiny treasury. The goal is to prove that
the revenue control plane, treasury policy, swap lifecycle, accounting, and repair paths work
correctly with real money under tight limits.

## Scope

This exercise validates:

- treasury seeding and visibility
- operator-configured swap policy
- revenue opportunity lifecycle
- one low-cost earning path
- post-settlement swap queue, submission, and reconciliation
- profit/tax accounting
- mechanic repair visibility

This exercise explicitly does **not** attempt to validate:

- sustained autonomous earnings durability
- repeated or high-volume on-chain operations
- aggressive reinvestment behavior
- speculative or trading-oriented strategies

## Budget

Assumed total seed: **$50**

Recommended allocation:

- **$30** working stable balance
- **$15** gas / execution buffer
- **$5** contingency headroom

Abort if the remaining balance drops below the configured treasury minimum reserve.

## Preconditions

- wallet address visible from API, dashboard, and CLI
- treasury policy configured and understood
- revenue swap policy configured
- at least one EVM swap target configured with contract addresses
- mechanic reports no unresolved revenue control-plane issues
- operator confirms the active chain and RPC are intentional

## Phase 1: Seed And Visibility

1. Fund the treasury with the planned seed.
2. Verify:
   - `ironclad wallet`
   - dashboard wallet view
   - `GET /api/wallet/balance`
3. Confirm:
   - minimum reserve
   - per-payment cap
   - revenue swap target symbol and default chain
   - tax policy state

Pass criteria:

- balances agree across operator surfaces
- treasury controls are visible and coherent
- no unexpected transaction records appear

## Phase 2: Controlled Revenue Path

Use one intentionally cheap path:

- a minimal paid service, or
- a manually injected low-risk revenue opportunity through the canonical lifecycle

Run:

1. intake
2. qualify
3. plan
4. fulfill
5. settle

Pass criteria:

- opportunity state progresses legally
- settlement is idempotent
- revenue ledger entries appear
- gross, attributable cost, net profit, retained earnings, and tax all render correctly

## Phase 3: Swap Execution Validation

Use the settled opportunity from Phase 2.

Run:

1. verify swap task queued
2. submit swap
3. reconcile receipt
4. confirm final task state

Pass criteria:

- tx hash is recorded on submission
- reconcile path reports `pending`, `confirmed`, or `failed` honestly
- completed/failed state is folded back into the swap task
- swap transaction events are visible in the transaction ledger

## Phase 4: Failure And Repair Drill

Exercise one controlled failure case:

- missing contract address
- chain mismatch
- failed reconcile path

Then run:

- `ironclad mechanic`
- `ironclad mechanic --repair`

Pass criteria:

- the failure is explainable from operator surfaces
- mechanic reports the issue coherently
- repair does not corrupt revenue state

## Abort Conditions

Stop the exercise if any of the following occur:

- treasury minimum reserve would be violated
- swap submission attempts target the wrong chain
- accounting surfaces disagree materially
- mechanic cannot explain or repair resulting state
- a transaction is submitted without a clear operator explanation of why

## Success Criteria

`v0.9.6` passes the constrained live-money validation when:

- real funds can be seeded and observed
- one earning path completes through the canonical lifecycle
- one real post-settlement swap is submitted and reconciled
- accounting remains internally consistent
- mechanic can observe and repair the relevant control-plane state

Anything beyond that belongs to the later durability/viability exercise, not this seed test.
