# v0.9.4 Release Checklist (Advanced Routing Without ML-by-Default)

This release follows `v0.9.3` and focuses on improving routing quality/cost outcomes while ML remains in an isolated validation track.

## Scope (10 items)

1. Accuracy-target routing (`2.2`) for bounded per-request quality floors
2. Multi-provider cost arbitrage (`2.6`) in production ranking path
3. Routing feature schema versioning and trace completeness
4. Historical routing dataset extraction pipeline
5. Offline routing evaluation harness (quality/cost/latency regression checks)
6. Decision attribution (`metascore`, `fallback`, `arbitrage-adjusted`)
7. Shadow-evaluation pipeline for ML recommendations (no execution impact)
8. Canary control plane for future ML trials (disabled by default)
9. Rollback/kill-switch automation for routing strategy changes
10. Operator diagnostics for counterfactual and rollout analytics

## ML Policy for v0.9.4

- ML routing is **not shipped as default decision authority** in this release.
- ML work runs on feature branch `codex/ml-routing-validation-sandbox`.
- Promotion to product routing requires empirical proof from replay + shadow (+ optional canary) windows.
- Validation design source: `docs/plans/ml-routing-validation-sandbox.md`.

## Execution Order

### Phase 0: Routing Baseline Hardening

0a. **Feature schema + telemetry completeness**

- Finalize feature schema versioning and decision trace capture.
- Ensure all routing decisions are reproducible from logs.

0b. **Evaluation data pipeline**

- Build stable extraction of historical routing/outcome data.
- Define baseline regression thresholds and report format.

### Phase 1: Product Routing Improvements

1a. **Accuracy-target routing (`2.2`)**

- Support bounded quality targets.
- Enforce safe defaults and deterministic fallback behavior.

1b. **Cost arbitrage (`2.6`)**

- Normalize provider/model costs and add arbitrage signals to ranking.
- Keep policy/breaker/capacity as hard constraints.

### Phase 2: Validation Infrastructure for ML (non-shipping)

2a. **Shadow recommendation path**

- ML predicts alongside production routing; no user-visible control.
- Log counterfactual decisions and outcome deltas.

2b. **Canary control primitives (disabled by default)**

- Add traffic-split controls and kill switch.
- Keep runtime default on baseline routing.

### Phase 3: Safety, Rollback, and Operator Visibility

3a. **Rollback automation**

- Define and enforce rollback triggers for quality/cost/latency regression.

3b. **Operator diagnostics**

- Expose decision attribution and counterfactual metrics.
- Surface arbitrage/accuracy-target effects clearly.
