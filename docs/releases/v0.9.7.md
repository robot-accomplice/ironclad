# v0.9.7 Release Checklist — DB Fitness, Hardening & v1.0 Prep

Ironclad runs on an embedded SQLite database — a mal-performant data layer sinks the entire ship. The v0.9.6 DB health audit identified 18 concrete issues across 4 severity tiers. This release resolves all of them, adds operational hardening, and evaluates v1.0 readiness.

## Scope

Primary:

- **DB Fitness items 1–18** — SQLite performance hardening across all severity tiers
- **Sandbox Boundary Management** — Hardened execution boundaries for skill scripts
- **Memory Hygiene Mechanic** — Self-service detection and purge of contaminated memory entries

## Release Gates

### DB Fitness — CRITICAL (unbounded growth, data loss risk)

- [x] **DF-1**: Retention pruning for `turns`, `session_messages`, `tool_calls`, `inference_costs`, `model_selection_events`
- [x] **DF-2**: Working-memory orphan cleanup (periodic sweep for expired session references)
- [x] **DF-3**: Embedding orphan cleanup (periodic sweep for deleted parent rows)
- [x] **DF-4**: `auto_vacuum=INCREMENTAL` — deleted rows reclaim space

### DB Fitness — HIGH (missing indexes, query inefficiency)

- [x] **DF-5**: Index `turns.session_id`
- [x] **DF-6**: Index `working_memory.session_id`
- [x] **DF-7**: Episodic decay rerank rewrite (shipped in v0.9.6)
- [x] **DF-8**: Episodic dead-entry pruning for decayed-importance rows

### DB Fitness — MEDIUM (silent waste, stale infrastructure)

- [x] **DF-9**: `cron_runs` pruning + index on `job_id`
- [x] **DF-10**: `policy_decisions` indexes on `turn_id` and `created_at`
- [x] **DF-11**: `context_snapshots` retention pruning
- [x] **DF-12**: `shadow_routing_predictions` time-based retention
- [x] **DF-13**: `evict_expired_cache()` already wired (boot + 5-min timer in lib.rs:875)
- [x] **DF-14**: `semantic_cache` NULL expiry fix (default to now + 1 hour)

### DB Fitness — LOW (normalization, tuning, dead code)

- [x] **DF-15**: Dropped deprecated `embedding_json` column (blob-only storage)
- [x] **DF-16**: CHECK constraints on 11 low-cardinality TEXT columns
- [x] **DF-17**: `PRAGMA synchronous=NORMAL` under WAL (~2x write throughput)
- [x] **DF-18**: Dropped dead `proxy_stats` table

### Memory Hygiene Mechanic

- [x] `MemoryHygieneReport` struct and `run_memory_hygiene()` core logic in `helper_memory_hygiene.rs`
- [x] 7 deterministic LIKE-prefix patterns across 3 memory tiers (zero false-positive by design)
- [x] Diagnostic mode: `ironclad mechanic` shows per-tier contamination breakdown
- [x] Repair mode: `ironclad mechanic --repair` purges detected entries + VACUUM
- [x] JSON output: `MechanicFinding` with id `memory-contamination`, severity `high`, safe_auto_repair `true`
- [x] `RepairActionSummary.memory_entries_purged` field wired into repair flow
- [x] 4 unit tests covering detect-only, repair, clean-db, and missing-db paths
- [x] Feature branch `feat/mechanic-memory-hygiene` pushed and CI-green

### Sandbox Boundary Management

- [x] Filesystem confinement: skill scripts restricted to `skills_dir` subtree and explicit `$IRONCLAD_WORKSPACE` (no `..` traversal, no symlink escape)
- [x] Network policy: configurable network isolation for sandboxed scripts (deny-by-default opt-in via `network_allowed` config + Linux `unshare(CLONE_NEWNET)`)
- [x] Resource limits: wall-clock timeout enforcement hardened (existing `script_timeout_seconds`), memory ceiling via `RLIMIT_AS` on Linux (`script_max_memory_bytes` config, default 256 MiB)
- [x] Interpreter allowlist audit: `resolve_interpreter_absolute()` validates against absolute paths to prevent PATH hijacking; mechanic reports bare names
- [x] Mechanic check: `ironclad mechanic` reports sandbox configuration health (permissive interpreters, disabled sandbox, missing limits, platform limitations)
