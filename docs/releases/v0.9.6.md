# v0.9.6 Release Checklist ‚Äî Self-Funding Debut + Community Distribution

This release delivers the first complete self-funding revenue control plane, a skills catalog and registry protocol for community distribution, and native package manager installation via Homebrew and Winget.

## Scope

Primary:

- `2.18` Compliance-First Self-Funding Portfolio + Profit Taxation
- `2.14` Skills Catalog (CLI + Dashboard)
- `2.21` Skill Registry Protocol
- `1.20` Homebrew & Winget Package Manager Distribution

Complementary (shipped alongside):

- **Learning Loop Closure** ‚Äî procedural memory feedback loop (`learned_skills` table, `learning.rs`, governor wiring, skill priority reinforcement)
- **Progressive Context Compaction** ‚Äî 5-stage compaction with decay-weighted episodic retrieval
- **x402 Autonomous Payment** ‚Äî `402 Payment Required` handling with on-chain payment and retry

## Release Gates

### 2.18 Compliance-First Self-Funding

- [x] Revenue opportunity lifecycle is DB-backed and restart-safe: intake ‚Üí qualify ‚Üí score ‚Üí plan ‚Üí fulfill ‚Üí settle
- [x] Opportunity scoring persists `confidence_score`, `effort_score`, `risk_score`, `priority_score`, `recommended_approved`, and `score_reason`
- [x] Strategy outcome feedback is persisted per opportunity (`revenue_feedback` table) and summarized by strategy (`revenue_feedback_summary_by_strategy`, `revenue_feedback_signal_for_strategy`)
- [x] Post-settlement asset routing is operator-configurable (default `PALM_USD`) with contract address support
- [x] Queued EVM swaps can be submitted through the wallet with tx-hash tracking (`submit_evm_contract_call` ‚Üí `mark_revenue_swap_submitted`)
- [x] Submitted EVM swaps can be reconciled from on-chain receipts (`get_evm_transaction_receipt_status` ‚Üí `mark_revenue_swap_confirmed`/`mark_revenue_swap_failed`)
- [x] Tax payout tasks mirror the swap lifecycle: queue ‚Üí claim ‚Üí submit ‚Üí reconcile ‚Üí confirm/fail
- [x] Net profit, attributable costs, retained earnings, and tax allocation are observable from API (`revenue_accounting_summary`), CLI (`ironclad wallet`), and terminal mechanic surfaces
- [x] Mechanic can probe, reconcile, and repair orphaned or stale revenue jobs (`run_gateway_provider_and_revenue_checks`, `run_gateway_integrated_repair_sweep`)
- [x] No bespoke revenue path exists outside the canonical lifecycle ‚Äî all strategies flow through the shared control plane
- [x] Revenue strategy profitability and audit log (`revenue_strategy_profitability`, `revenue_audit_log`) surface cycle-time analytics and per-strategy conversion metrics

#### Seed Exercise ‚Äî Deferred

The `$50` seed exercise plan (`docs/releases/v0.9.6-seed-exercise.md`) validates the revenue infrastructure against real on-chain transactions. The exercise plan is authored and the rails are proven via integration tests, but **the exercise itself is deferred**: the economic ecosystem for autonomous bot services is still nascent ‚Äî no reliable counterparty market exists to exercise against. The infrastructure is ready; the market is not.

When the ecosystem matures sufficiently to support real paid-service execution, the seed exercise can be run as-written without code changes.
