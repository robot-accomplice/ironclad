# v0.8.2 Release Notes

Stabilization, test coverage hardening, and package distribution release. Resolves 29 bugs from v0.8.1 testing, adds 100+ integration tests, and ships Homebrew/Winget package manager support.

## Highlights

- **29 bug fixes** across input validation, API error handling, cron scheduler, dashboard, config, model persistence, and update subsystems
- **100+ new integration tests** covering all major API route categories (3,316 total workspace tests)
- **Homebrew tap** for macOS/Linux (`brew install robot-accomplice/tap/ironclad`)
- **Winget manifest** for Windows distribution

## Bug Details

### High Severity

#### BUG-001: Duplicate trailing content after `</html>` in dashboard SPA
- **File**: `crates/ironclad-server/src/dashboard_spa.html:6000-6013`
- **Description**: Lines 6000-6013 are a truncated duplicate of the WebSocket initialization block, appearing as raw text after the `</html>` close tag. Browsers silently ignore it, but it inflates the SPA payload and is a code hygiene issue.
- **Repro**: View source of `GET /` and inspect content after `</html>`.
- **Fix**: Remove duplicate lines 6000-6013.

#### BUG-009: Session `agent_id` stored without HTML sanitization
- **File**: `crates/ironclad-server/src/api/routes/sessions.rs:67`
- **Description**: `create_session` calls `validate_short()` (null byte + length check) but does not call `sanitize_html()` on `agent_id`. Raw `<script>` tags are stored in the database. The dashboard SPA escapes via `esc()` at render time, but the stored value is a latent XSS vector for any future rendering path (exports, emails, alternate frontends).
- **Repro**: `POST /api/sessions` with `{"agent_id":"<script>alert(1)</script>"}` → 200, stored raw.
- **Fix**: Apply `sanitize_html()` to `body.agent_id` before storage, matching sub-agent name handling.

#### BUG-011: Invalid cron expressions accepted without validation
- **File**: `crates/ironclad-server/src/api/routes/cron.rs:50-80`
- **Description**: Cron job creation accepts any string as `schedule_expr` — including `"NOT_VALID_CRON"`, `"*"`, or garbage. No cron expression parsing/validation occurs. These jobs will fail silently at runtime when the scheduler tries to compute `next_run_at`.
- **Repro**: `POST /api/cron/jobs` with `{"name":"test","schedule_kind":"cron","schedule_expr":"NOT_VALID","agent_id":"test","action":"test"}` → 200.
- **Fix**: Add cron expression validation (e.g., via the `cron` crate's `Schedule::from_str()`) before persisting.

#### BUG-022: Cron `schedule` field silently discarded; API expects `schedule_expr`
- **File**: `crates/ironclad-server/src/api/routes/cron.rs:11-17`
- **Description**: `CreateCronJobRequest` defines the field as `schedule_expr`, not `schedule`. Clients sending `{"schedule":"0 * * * *"}` have the field silently ignored by serde (unknown fields discarded). The job is created with `schedule_expr: null`, which means it never runs.
- **Repro**: `POST /api/cron/jobs` with `"schedule":"0 * * * *"` → 200, but `GET` shows `schedule_expr: null`.
- **Fix**: Either (a) add `#[serde(alias = "schedule")]` to `schedule_expr`, or (b) add `#[serde(deny_unknown_fields)]` to `CreateCronJobRequest` so unknown fields produce an error, or (c) document the correct field name prominently.

#### BUG-026: Model changes via dashboard not persisted to disk
- **File**: `crates/ironclad-server/src/api/routes/admin.rs:1712-1805`
- **Description**: `change_agent_model` for the commander agent updates in-memory config and syncs the router but never persists the change to `ironclad.toml`. The response even documents this: `"scope": "commander (runtime only, not persisted to disk)"`. After server restart, the old model loads from the config file, negating the user's model selection.
- **Repro**: Change model via dashboard Apply button → restart server → model reverts to config file value.
- **Fix**: Persist the model change to `ironclad.toml` when updating the commander agent's primary model.

#### BUG-031: TOML config serialization breaks on Windows paths with backslashes
- **File**: `crates/ironclad-server/src/api/routes/admin.rs` (config write path)
- **Description**: When config is serialized back to TOML on Windows, paths like `C:\Users\jonat\...` produce invalid TOML because `\U` is interpreted as a unicode escape sequence in basic (double-quoted) strings. The TOML spec requires 8-digit hex after `\U`, but the raw path has `\U` followed by `sers`.
- **Repro**: On Windows, add a model in settings and save → `{"error":"TOML parse error: ... invalid unicode 8-digit hex code"}`.
- **Fix**: Ensure paths are serialized using TOML literal strings (single-quoted) that don't interpret escape sequences, or normalize to forward slashes before writing.

### Medium Severity

#### BUG-002: No WebSocket reconnection logic in dashboard
- **File**: `crates/ironclad-server/src/dashboard_spa.html:5960`
- **Description**: The `onclose` and `onerror` handlers only update the connection indicator dot. No reconnection attempt is made. After a server restart, users must manually reload the page to restore real-time updates.
- **Fix**: Add exponential backoff reconnection in `onclose` handler.

#### BUG-003: Sessions list hardcoded `LIMIT 200` with no pagination
- **File**: `crates/ironclad-server/src/api/routes/sessions.rs:37`
- **Description**: `list_sessions` query uses `LIMIT 200` hardcoded. No offset/cursor parameter is supported. Installations with >200 sessions cannot access older entries via the API.
- **Fix**: Accept `offset` query parameter and use client-provided `limit` (clamped to a reasonable max).

#### BUG-004: `validate_short()` does not reject empty strings
- **File**: `crates/ironclad-server/src/api/routes/mod.rs:148-164`
- **Description**: `validate_short()` checks null bytes and max length but allows empty strings. Empty `agent_id` on sessions or empty `name` on cron jobs both pass validation and are persisted.
- **Repro**: `POST /api/sessions` with `{"agent_id":""}` → 200.
- **Fix**: Add `if val.is_empty() { return Err(bad_request(...)); }` to `validate_short()` and `validate_long()`.

#### BUG-006: Malformed JSON / wrong Content-Type errors return `text/plain`
- **File**: `crates/ironclad-server/src/api/routes/mod.rs` (axum rejection layer)
- **Description**: Sending invalid JSON or wrong Content-Type to POST endpoints returns plain text errors (e.g., `"Expected request with Content-Type: application/json"`) instead of structured `{"error":"..."}` JSON. This breaks API clients that expect consistent JSON error format.
- **Repro**: `POST /api/sessions` with `Content-Type: text/plain` → 415 with plain text body.
- **Fix**: Add custom axum rejection handler that wraps all rejection responses in `{"error":"..."}` JSON.

#### BUG-007: Negative `limit` parameter returns all rows
- **File**: `crates/ironclad-server/src/api/routes/sessions.rs`, `memory.rs`, and other list endpoints
- **Description**: `?limit=-1` is passed directly to SQLite `LIMIT` clause. SQLite treats negative LIMIT as "no limit", returning all rows. This bypasses intended pagination and could be used to dump large datasets.
- **Repro**: `GET /api/sessions?limit=-1` returns all sessions. Same on memory endpoints.
- **Fix**: Clamp `limit` to `1..=max` range (e.g., max 200) before SQL query.

#### BUG-008: Huge `limit` parameter accepted without clamping
- **File**: Same endpoints as BUG-007.
- **Description**: `?limit=999999` is accepted and passed to SQL. While the database handles it, the response payload could be very large, causing memory pressure.
- **Fix**: Clamp `limit` to a reasonable maximum (e.g., 500) with the same fix as BUG-007.

#### BUG-010: Empty cron job `name` accepted
- **File**: `crates/ironclad-server/src/api/routes/cron.rs`
- **Description**: Cron job creation accepts `"name":""`. An empty-named cron job is persisted and appears in listings.
- **Repro**: `POST /api/cron/jobs` with `{"name":"","schedule_kind":"cron","schedule_expr":"* * * * *","agent_id":"test","action":"test"}` → 200.
- **Fix**: Validate `name` is non-empty in cron job creation handler.

#### BUG-012: Invalid `schedule_kind` accepted without validation
- **File**: `crates/ironclad-server/src/api/routes/cron.rs:223-228`
- **Description**: `normalize_schedule_kind()` only maps `"interval"` → `"every"`. All other values (including `"invalid"`, `"banana"`, etc.) pass through unchanged. No validation against allowed kinds (`cron`, `every`).
- **Fix**: Reject `schedule_kind` values that are not `"cron"` or `"every"` (after normalization).

#### BUG-013: Invalid `payload_json` accepted without validation
- **File**: `crates/ironclad-server/src/api/routes/cron.rs`
- **Description**: Cron job `payload_json` accepts any string, including non-JSON text like `"not json at all"`. The field is stored as-is with no JSON validity check.
- **Fix**: If field is named `payload_json`, validate it parses as valid JSON before storing.

#### BUG-014: Missing required cron fields returns `text/plain` 422
- **File**: `crates/ironclad-server/src/api/routes/cron.rs` (serde deserialization)
- **Description**: Sending a cron job creation request with missing required fields returns a plain text 422 error from serde, not structured JSON. Inconsistent with the `{"error":"..."}` format used elsewhere.
- **Fix**: Same custom rejection handler as BUG-006.

#### BUG-016: Missing message `role` returns `text/plain` 422
- **File**: `crates/ironclad-server/src/api/routes/sessions.rs` (serde rejection)
- **Description**: Same issue as BUG-014 — serde deserialization errors on session message creation return plain text, not JSON.
- **Fix**: Same custom rejection handler as BUG-006.

#### BUG-018: No `Content-Security-Policy` response header
- **File**: `crates/ironclad-server/src/api/routes/mod.rs` (middleware layer)
- **Description**: The server does not send a `Content-Security-Policy` header on any response. The dashboard SPA uses inline scripts and styles, so at minimum a CSP with appropriate `script-src` and `style-src` directives should be set.
- **Fix**: Add CSP header via middleware. Start with `default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'` and tighten over time.

#### BUG-019: No `X-Frame-Options` response header
- **File**: Same as BUG-018.
- **Description**: No `X-Frame-Options` or `frame-ancestors` CSP directive is set. The dashboard could be embedded in an iframe for clickjacking attacks.
- **Fix**: Add `X-Frame-Options: DENY` header (or `frame-ancestors 'none'` in CSP).

#### BUG-020: `ironclad update` download method silently falls back to cargo build
- **File**: `crates/ironclad-server/src/cli/update.rs:655-687`
- **Description**: When `apply_binary_update` is called with `method="download"` and the download fails, it silently falls back to `apply_binary_cargo_update()` — especially with the `--yes` flag, which auto-confirms the fallback without user awareness. The user expects pre-compiled binaries but gets a local cargo build instead.
- **Repro**: Run `ironclad update --yes` when the download URL is unreachable.
- **Fix**: Make the fallback explicit — either require `--allow-build-fallback` flag, or always prompt (ignoring `--yes` for this specific decision).

#### BUG-021: Sub-agent `status` field is phantom — no database column
- **File**: `crates/ironclad-server/src/api/routes/subagents.rs`, DB schema `sub_agents` table
- **Description**: The sub-agent API accepts and returns a `status` field, but the `sub_agents` database table has no `status` column. The field is always `null` in responses. Clients may set `status` values that appear accepted but are silently discarded.
- **Fix**: Either (a) add a `status` column to the `sub_agents` table with a migration, or (b) remove the `status` field from the API response struct to avoid confusion.

#### BUG-023: Config PUT accepts arbitrary unknown section names
- **File**: `crates/ironclad-server/src/api/routes/admin.rs:252-254, 655-685`
- **Description**: `UpdateConfigRequest` uses `#[serde(flatten)]` to accept any JSON, then deep-merges it into the config. Unknown top-level keys (e.g., `{"__hack__":{"evil":"payload"}}`) are silently ignored during `IroncladConfig` deserialization, but the API returns `{"updated":true}`. This is misleading — the API claims success even when the patch had zero effect.
- **Fix**: Either (a) validate that all top-level keys in the patch are known config sections before merging, or (b) return a response indicating which fields were actually applied vs. ignored.

#### BUG-027: WebSocket `agent_working` event hardcodes `agent_id: "ironclad"`
- **File**: `crates/ironclad-server/src/api/routes/agent.rs:2206`
- **Description**: The `agent_working` WebSocket event published during streaming inference hardcodes `"agent_id": "ironclad"` instead of using the actual configured agent name/id. Dashboard consumers relying on this event receive incorrect agent identity.
- **Repro**: Open WebSocket → send message via `/api/agent/message/stream` → observe `agent_working` event contains `"agent_id": "ironclad"` regardless of configured agent.
- **Fix**: Replace hardcoded `"ironclad"` with the actual agent ID from config.

#### BUG-028: Skills page shows doubled "built-in" badge and grayed-out toggle
- **File**: `crates/ironclad-server/src/dashboard_spa.html:1807-1817`
- **Description**: Built-in skills render with TWO separate "built-in" badges (one small muted badge and one adjacent name tag) plus a grayed-out disabled toggle switch. The combination makes built-in skills look deactivated when they are always active.
- **Repro**: Navigate to Skills page in dashboard → observe built-in skills appear turned off.
- **Fix**: Remove the toggle switch for built-in skills entirely and replace with a single green "BUILT-IN" badge that clearly communicates always-active status.

#### BUG-029: Dashboard checks `agent.name` but status API returns `agent_name`
- **File**: `crates/ironclad-server/src/api/routes/agent.rs:1073-1088`, `dashboard_spa.html:1377`
- **Description**: The `/api/agent/status` endpoint returns `agent_name` as the field name, but the dashboard JavaScript checks `agent.name`. Since `agent.name` is undefined, the agent display name falls through to the health endpoint fallback instead of being set directly from the status response.
- **Repro**: Load dashboard → observe `agent.name` is undefined in status response, display name comes from health fallback.
- **Fix**: Add `"name": config.agent.name` to the status response alongside `agent_name` for backward compatibility.

### Low Severity

#### BUG-005: Inference costs hardcoded `LIMIT 100` with no pagination
- **File**: `crates/ironclad-server/src/api/routes/admin.rs:723`
- **Description**: Cost history query is hardcoded to `LIMIT 100`. No offset parameter. Consistent with BUG-003 pattern but lower impact since cost records are less frequently accessed.
- **Fix**: Accept `offset` + `limit` (clamped) query parameters.

#### BUG-015: Empty message content accepted
- **File**: `crates/ironclad-server/src/api/routes/sessions.rs`
- **Description**: Session message creation accepts `"content":""`. While technically valid (some protocols use empty messages as signals), it may cause display issues in the dashboard chat view.
- **Fix**: Consider rejecting empty content with a validation check, or document as intentional.

#### BUG-017: 405 Method Not Allowed returns empty body
- **File**: `crates/ironclad-server/src/api/routes/mod.rs` (axum default handler)
- **Description**: Sending an unsupported HTTP method (e.g., `DELETE /api/sessions`) returns 405 with an empty response body. API clients expect a JSON error body.
- **Fix**: Add a custom 405 handler that returns `{"error":"method not allowed"}`.

#### BUG-024: Approvals approve/deny endpoints require `Content-Type: application/json` with empty body
- **File**: `crates/ironclad-server/src/api/routes/admin.rs` (approvals handlers)
- **Description**: `POST /api/approvals/{id}/approve` and `/deny` return 415 if no `Content-Type: application/json` header is sent, even though the handler doesn't need a request body. This is unusual for action endpoints that take no input.
- **Fix**: Either (a) remove the JSON extractor since no body is needed, or (b) document the requirement.

#### BUG-030: Model list in dashboard doesn't highlight newly detected models
- **File**: `crates/ironclad-server/src/dashboard_spa.html` (model selection UI)
- **Description**: When a new model becomes available (e.g., freshly installed via ollama), the model list does not sort or highlight it. Users must scroll through the entire model list to find the new model.
- **Fix**: Sort recently detected / newly available models to the top of the list, or add a "New" indicator.

#### BUG-025: Config PUT returns `"updated":true` even when no effective changes were made
- **File**: `crates/ironclad-server/src/api/routes/admin.rs:655-685`
- **Description**: Related to BUG-023. Even when the config patch contains only unknown keys that are silently dropped, or identical values to the current config, the response is `{"updated":true,"persisted":true}`. The backup file is also created unnecessarily.
- **Fix**: Compare pre- and post-merge config values; only report `"updated":true` and create backups when actual changes occurred.
