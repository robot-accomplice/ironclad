# v0.9.1 Release Checklist (Intelligent Routing)

This release transforms the model selection hot path from availability-first (first usable model wins) into an intelligent, data-driven pipeline. ~1,838 lines of dead routing infrastructure (`tiered.rs`, `uniroute.rs`, `router.rs`, `accuracy.rs`) get wired into production. The agent will select models based on task complexity, cost, quality history, and capability matching instead of static config order.

## Implementation Notes

### Routing Hot Path Transformation

Current (`select_routed_model_with_audit()`):
```
config order ŌåÆ first usable ŌåÆ return
```

Target:
```
user_content ŌåÆ extract_features ŌåÆ classify_complexity
  ŌåÆ metascore(candidates, task_requirements)
  ŌåÆ select_for_complexity (or select_cheapest_qualified if budget-constrained)
  ŌåÆ confidence evaluation on response (if tiered enabled)
  ŌåÆ escalation if below confidence floor
  ŌåÆ persist audit with full scoring context
```

### Quality Signal Bootstrap Problem

`QualityTracker` needs observation data to be useful, but starts empty. Bootstrap strategy:
- Seed initial quality priors from model profile config (operator-defined)
- Use heuristic quality signal for first N turns (response length, structure, finish_reason)
- Decay toward observed quality as samples accumulate (exponential moving average)

### 2.19 / 2.3 Ordering

Wire 2.19 (metascore routing) BEFORE 2.3 (tiered inference) because:
- Metascore provides the foundation for deciding which tier to try first
- Tiered inference adds complexity (confidence evaluation, escalation) that benefits from having a working quality tracker
- Both share the routing hot path ŌĆö doing metascore first means tiered inference can build on the new structure
