# v0.9.x Release Series — Infrastructure-First Plan

This document defines the v0.9.0 through v0.9.7 release series. The guiding principle is **infrastructure before features**: land the big architectural changes early so that dependent features can iterate rapidly on a stable foundation.

## Design Rationale

After four consecutive bug-fix releases (v0.8.6–v0.8.9, ~180 fixes), the remaining high-impact issues are architectural:

1. **Dual-path divergence** — `agent_message` and `process_channel_message` require manual synchronization, causing recurring bugs (BUG-3/7/11 class).
2. **Stale context dredging** — Memory retrieval lacks a relevance/recency gate, so the agent surfaces resolved references.
3. **Dead routing infrastructure** — `ModelVector`, `QualityTracker`, `select_for_complexity()` exist as dead code while the live hot path ignores complexity scoring.
4. **Cold boot latency** — Full context rebuild from DB on every boot.
5. **No agent self-awareness** — Channel context, memory stats, and health metrics require system prompt injection instead of on-demand tool calls.

These are not patchable; they require the architectural work planned below.

## Release Details

### v0.9.0 — Foundation

**Theme**: Lay the architectural foundation.

See `docs/releases/v0.9.0.md` for full release gates.

| Item | Effort | Rationale |
|---|---|---|
| Single-pathway consolidation | M | Eliminates dual-path divergence; every subsequent feature benefits |
| 1.11 Context Checkpoint | M | Instant boot, crash recovery, faster dev cycles |
| 1.16 Durable Delivery Queue | M | Crash-safe channels; prerequisite for all channel work |
| 1.22 Introspection Skill | S | `get_runtime_context`, `get_memory_stats`, `get_channel_health` + `ToolContext` channel extension; zero-cost self-awareness eliminates 1.23 stopgap |
| 2.12 Episodic Digest | M | Fixes stale-context dredging; improves all downstream quality |
| 2.8 Prompt Compression | L | Token cost reduction across all features |
| 1.18 Cron Rotation (stretch) | S | Correctness fix, small scope |

### v0.9.1 — Intelligent Routing

**Theme**: Unified, data-driven model selection with tiered inference.

| Item | Effort | Rationale |
|---|---|---|
| 2.19 Metascore Routing (core) | L | Resurrects dead routing code; unified profile-based selection replaces scattered heuristics |
| 2.3 Tiered Inference Pipeline | M | Wires existing dead code (ConfidenceEvaluator, EscalationTracker) into production; cache→local→cloud escalation path |
| 1.17 Production Abuse Protection | M | Required before internet-facing deployment; roadmap specifies v0.9.1 |

**Scope note**: 2.19 dashboard panels (explainability + simulation) deferred to v0.9.2 per cut policy. Core scope: `ModelProfile` schema, `TaskRequirements`, `metascore()` computation, routing integration, persistence. 2.3 shares the routing hot path with 2.19 — co-locating avoids refactoring `select_routed_model_with_audit()` twice.

**Dropped**: `1.23` Context Budget Tuning — superseded by 1.22 introspection in v0.9.0. The stopgap L0 raise is no longer needed when the agent can query its own context budget on demand.

---

### v0.9.2 — Operator Visibility & Data Architecture

**Theme**: Make the system observable, operable, and self-aware about its own data.

Absorbs the original v0.9.3 (Operator Tooling) and v0.9.4 (Data Architecture) scopes. The 1.22 introspection extensions originally planned here are deferred until real usage patterns from v0.9.0/v0.9.1 are available.

| Item | Effort | Rationale |
|---|---|---|
| 2.19 dashboard panels + explainability | M | Metascore breakdown, candidate comparison, simulation mode, routing decision trace in Context Observatory |
| 1.21 Integrations Management (CLI) | M | `ironclad integrations list/test/connect/disconnect` — CLI-first, dashboard follows |
| 2.15 Ops Telemetry + Provenance | M | Release smoke gates, deployment readiness checks, artifact provenance |
| 2.13 Hippocampus | L | Self-describing schema map; agent-created tables with guardrails; ambient storage awareness |
| 3.5.5 Document Ingestion | M | PDF/markdown/code file ingestion into memory system; extends knowledge beyond conversation |
| 1.18 Cron Rotation | S | Correctness fix deferred from v0.9.0 stretch; overdue |
| `inference_costs` extension | S | Add `latency_ms`, quality columns — feeds metascore weight tuning |

**Scope note**: 1L + 4M + 2S. Dashboard explainability and Hippocampus are the two anchor items. The inference_costs extension enriches both: dashboard panels can visualize latency/quality trends, and Hippocampus schema map includes the extended columns in its self-description.

**Deferred**: 1.22 introspection extensions — revisit when v0.9.0/v0.9.1 usage patterns are available.

### v0.9.4 — Advanced Routing

**Theme**: Cost-aware routing improvements with ML validation-first (non-default).

| Item | Effort | Rationale |
|---|---|---|
| 2.1 ML-Based Routing (validation track) | L | Long-running replay/shadow/canary validation branch; not default product authority until empirical promotion gates pass |
| 2.2 Accuracy-Target Routing | M | Per-request quality targets (τ); Lagrangian cost optimization |
| 2.6 Cost Arbitrage | M | Real-time pricing awareness; cheapest provider meeting quality floor |

**Dependency**: All three build on 2.19 metascore infrastructure and require quality/cost telemetry from `inference_costs`.

**ML policy for v0.9.4**: ML routing remains in an isolated validation branch (`codex/ml-routing-validation-sandbox`) using replay/shadow evaluation. `v0.9.4` ships with baseline routing authority unless promotion criteria are met.

### v0.9.6 — Self-Funding Debut + Community Distribution

**Theme**: First complete autonomous revenue loop, with ecosystem growth and zero-friction installation.

| Item | Effort | Rationale |
|---|---|---|
| 2.18 Compliance-First Self-Funding | L | Debut the first full self-funding mechanism end-to-end rather than leaving revenue in scaffold form |
| 2.14 Skills Catalog | M | CLI + dashboard browse/install for first-party skills |
| 2.21 Skill Registry Protocol | M | Community distribution without centralized marketplace |
| 1.20 Homebrew & Winget | S | `brew install ironclad`, `winget install RobotAccomplice.Ironclad` |

**Exit criteria for v0.9.6**:

- at least one paid service strategy and one non-service strategy run through the same revenue control plane
- opportunity intake, qualification, scoring, planning, fulfillment, settlement, and feedback are all DB-backed and restart-safe
- opportunity scoring persists confidence, effort, risk, priority, and recommendation fields and is visible to operator surfaces
- strategy outcome feedback is persisted per opportunity and summarized by strategy
- settlement supports configurable post-settlement asset routing (default `PALM_USD`) with operator-controlled swap disable/override behavior
- queued EVM swaps can be submitted through the wallet on the configured chain with tx-hash tracking
- net profit, attributable costs, retained earnings, and tax allocation are observable from API, dashboard, and terminal surfaces
- mechanic can probe, reconcile, and repair orphaned or stale revenue jobs
- no bespoke revenue path exists outside the canonical lifecycle

## Governance

Each v0.9.x release:

1. Writes its own `docs/releases/v0.9.X.md` release checklist before implementation begins.
2. Runs `just ci-test` with no coverage regression from the prior release baseline.
3. Updates `docs/ROADMAP.md` statuses for completed items.
4. Updates CHANGELOG.md with a versioned entry.
5. Evaluates whether stretch items from the prior release should be absorbed or re-deferred.
