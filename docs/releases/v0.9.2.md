# v0.9.2 Release Checklist (Wiring Remediation + Operator Visibility & Data Architecture)

This release addresses critical wiring gaps identified by the comprehensive wiring audit
(`docs/audit/wiring-audit-v0.9.md`), then delivers operator visibility and data architecture
features. The wiring remediation is Phase 0 — it must land first because downstream features
depend on foundations that are currently broken or disconnected.

## Scope

**Wiring Remediation (12 items — Phase 0):**

- Tier 1 (Immediate): Dual-path consolidation, `post_turn_ingest` tool_results fix, API-path gate system note
- Tier 2 (High Value): Multi-tool parsing, OpenAiResponses/Google tool support, quality feedback loop, approval resume
- Tier 3 (Cleanup): SpawnManager removal, dead routing code removal, `decay_importance` wiring, context pruning wiring, checkpoint load wiring

**Features (7 items — Phases 1–6, unchanged):**

- `2.19` Dashboard panels + explainability — metascore breakdown, candidate comparison, simulation mode
- `1.21` Integrations Management (CLI) — `ironclad integrations list/test/connect/disconnect`
- `2.15` Ops Telemetry + Provenance — release smoke gates, deployment readiness, artifact provenance
- `2.13` Hippocampus — self-describing schema map, agent-created tables with guardrails
- `3.5.5` Document Ingestion Pipeline — PDF/markdown/code → memory system
- `1.18` Cron Rotation — cron-conformant session reset behavior
- `inference_costs` extension — add `latency_ms`, `quality_score`, `escalation` columns

Deferred:

- `1.22` introspection extensions — revisit when v0.9.0/v0.9.1 usage patterns are available
- Wiring audit Tier 4 (new capabilities) — abuse protection (1.17), MCP client discovery, cron→inference bridge, voice receive path → **v0.9.3**

## Execution Order

### Phase 0: Wiring Remediation

Source: `docs/audit/wiring-audit-v0.9.md` Tiers 1–3. Tier 4 deferred to v0.9.3.

**Tier 1 — Immediate (blocks current functionality)**

0a. **Dual-path consolidation** — Merge `agent_message` (API) and `process_channel_message`
    (channel) into a single `process_request` pipeline. Both entry points funnel into the
    same code path for cache check, diagnostics injection, gate system note, tool definitions,
    prepare_inference, and response caching. Eliminates 6+ behavioral asymmetries documented
    in the audit (Section 4.2). This is the single highest-impact wiring fix.

0b. **Fix `post_turn_ingest` tool_results** — All 3 call sites in `core.rs` pass `&[]` for
    `tool_results`. Change to pass the actual tool call name + result from the ReAct loop.
    Without this, episodic memory never captures tool-use context, degrading digest quality
    and causing the "stale context dredging" bug.

0c. **Wire `build_gate_system_note` in API path** — Currently only the channel path calls
    `build_gate_system_note()`. Add the call in `handlers.rs` so API-originated requests
    also receive delegation guidance. (Becomes moot if 0a lands first, but listed for
    independent fixability.)

**Tier 2 — High Value (enables planned features)**

0d. **Multi-tool parsing** — `parse_openai_completions_response` only takes
    `tool_calls.first()`, silently dropping additional tool calls. Parse all tool calls
    and concatenate into the text response. Same for Anthropic multi-content-block responses.

0e. **OpenAiResponses + GoogleGenerativeAi tool support** — Add `tools` parameter passthrough
    in `translate_openai_responses` and `translate_google`. Add structured tool call parsing
    in `parse_openai_responses_response` and `parse_google_response`. Also fix
    OpenAiResponses system prompt flattening and Google system message filtering.

0f. **Quality feedback loop** — On LlmService startup, load recent `inference_costs` rows
    and seed `QualityTracker` with historical observations. Record the richer confidence
    score (not just binary 0.5/1.0). Use the shared `ConfidenceEvaluator` on LlmService
    instead of creating a local copy in `infer_with_fallback`. Read `EscalationTracker`
    history to inform routing weight adjustments.

0g. **Approval resume** — After an approval is granted for a blocked tool call, queue
    re-execution of that tool call. Currently approval blocks but has no continuation
    mechanism.

**Tier 3 — Cleanup (remove dead code or wire it)**

0h. **Remove `SpawnManager`** — Complete dead code (zero server references). The virtual
    delegation tool pattern replaced it. Remove `spawning.rs` and the `spawn_manager` field
    from AppState.

0i. **Remove dead routing code** — Remove `ModelRouter` (`router.rs`), entire `uniroute.rs`
    module (`ModelVector`, `QueryRequirements`, `ModelVectorRegistry`), and dead functions
    `select_for_complexity()`, `select_cheapest_qualified()`, `select_for_quality_target()`.
    Production uses `select_by_metascore()` exclusively.

0j. **Wire `decay_importance`** — Call from `SessionGovernor.tick()` after `digest_on_close`.
    Older memories decay, preventing the stale context accumulation problem.

0k. **Wire context pruning** — In `build_context`, call `needs_pruning()` → `soft_trim()`
    when the assembled context exceeds the token budget. Currently the pruning functions
    exist but are never called, so context silently overflows.

0l. **Wire `load_checkpoint`** — On session resume, call `load_checkpoint` to restore
    prior context state. Currently `save_checkpoint` fires but load is never called,
    making checkpoints write-only.

### Phase 1: Schema & Data Foundation

1. **`inference_costs` schema extension** — Add `latency_ms` (INTEGER), `quality_score` (REAL nullable), `escalation` (BOOLEAN) columns to existing `inference_costs` table. Add migration. Update `persist_inference_cost()` call sites in LlmService to record latency and escalation flag. Back-fill quality_score from QualityTracker when available.

2. **1.18 Cron Rotation** — Evaluate session reset schedules directly from configured cron expressions. Add conformance tests for edge schedules (DST transitions, sub-minute cron, timezone-prefixed expressions). Document migration notes if semantics differ from legacy timing behavior.

3. **Hippocampus table + bootstrap** — Create `hippocampus` table: `table_name TEXT PK`, `owner TEXT` (system | agent_id), `schema_ddl TEXT`, `description TEXT`, `query_patterns TEXT`, `relationships TEXT`, `access_level TEXT` (read_only | read_write | internal), `row_count INTEGER`, `last_modified TEXT`. Populate from all system tables in `initialize_db()`. Add consistency check on boot.

### Phase 2: Agent Data Tools

4. **`CreateTable` / `AlterTable` / `DropTable` tools** — Register in ToolRegistry. Agent-created tables prefixed with agent id (e.g., `ag_duncan_research_notes`). Enforce: no reserved column names, no system table collisions, max tables per agent, max total size. Gate behind `RiskLevel::Caution`. Auto-register in hippocampus.

5. **Hippocampus context injection** — In `build_context()`, inject compact hippocampus summary (table names + one-line descriptions) into system prompt. Budget: ~200 tokens for the summary. Agent gains ambient awareness of its storage capabilities.

6. **3.5.5 Document Ingestion** — CLI: `ironclad ingest <path>` for PDF/markdown/code files. Parse → chunk (512 tokens, 64-token overlap using existing chunking infra) → embed → store in memory system. Support: `.md`, `.txt`, `.rs`, `.py`, `.js`, `.ts`, `.pdf` (via lightweight parser). API: `POST /api/knowledge/ingest` with multipart upload. Register ingested documents in hippocampus as knowledge sources.

8. **Integrations API** — `GET /api/integrations` returning per-channel status. `POST /api/integrations/{channel}/test` for live probes. Wire into existing `ChannelRouter` health metrics.

## Implementation Notes

### Hippocampus Design Principles

The hippocampus is the agent's "awareness of self-storage". Key constraints:
- Agent-created tables are sandboxed: prefixed, size-limited, no PRAGMA/ATTACH access
- System tables are read-only from the agent's perspective
- The hippocampus summary in context is compact (~200 tokens) — full schema details are available via introspection tools
- Stats (row_count, last_modified) refresh on MetricSnapshot heartbeat, not on every query

### Document Ingestion Architecture

Builds on completed RAG infrastructure (3.5 items 1–4):
- Binary embedding storage (done)
- HNSW ANN index (done)
- Content chunking — 512 tokens, 64-token overlap (done)
- Persistent semantic cache (done)

New work: file parsing → chunking → embedding → storage pipeline. PDF parsing should be lightweight (no heavy C dependencies) — consider `pdf-extract` or similar pure-Rust crate.

### Integrations CLI UX

The `connect` wizard should be channel-aware:
- Telegram: prompts for bot token, validates with getMe API
- WhatsApp: prompts for phone number ID + access token, validates with profile API
- Discord: prompts for bot token, validates with gateway identify
- Slack: prompts for bot token + signing secret, validates with auth.test

Each adapter's `ChannelAdapter` trait already has a health check method — the CLI `test` command calls it directly.

### Simulation Mode

The simulation endpoint reuses `build_model_profiles()` + `select_by_metascore()` but skips actual inference. It runs the full scoring pipeline against a synthetic task with operator-provided complexity hints and weight overrides, returning the same `MetascoreBreakdown` struct the real routing path produces.
