### CRITICAL Bugs (16)

#### BUG-300: Keystore deadlock — double mutex lock ordering
- **Crate**: `ironclad-core`
- **File**: `crates/ironclad-core/src/keystore.rs`
- **Description**: Two code paths acquire the keystore mutex and an inner encryption mutex in different orders. Under concurrent access (e.g., key rotation + read), this creates a classic ABBA deadlock. The process hangs permanently with no timeout.
- **Impact**: Complete agent freeze requiring process kill.
- **Fix**: Establish a single lock ordering convention; or collapse into a single mutex.

#### BUG-301: UTF-8 panic on multi-byte boundary in assistant memory truncation
- **Crate**: `ironclad-core`
- **File**: `crates/ironclad-core/src/memory.rs`
- **Description**: When truncating assistant messages to fit context budget, the code slices at a byte offset that can land in the middle of a multi-byte UTF-8 character (e.g., emoji, CJK). This causes a panic on `&content[..max_bytes]`.
- **Impact**: Agent crash on any conversation containing non-ASCII text near the truncation boundary.
- **Fix**: Use `content.char_indices()` to find the nearest valid character boundary before slicing.

#### BUG-302: Financial policy bypass for float-valued amounts
- **Crate**: `ironclad-core`
- **File**: `crates/ironclad-core/src/policy.rs`
- **Description**: Treasury policy uses `f64` comparison for spend limits. Values like `99.999999999999999` compare as equal to `100.0` due to IEEE 754 precision, allowing the policy check to pass when the actual amount exceeds the limit by a small epsilon. All financial arithmetic uses `f64` throughout the treasury module.
- **Impact**: Policy bypass for amounts near limit boundaries; potential for accumulated rounding drift in yield calculations.
- **Fix**: Use integer arithmetic in smallest denomination (wei/satoshi/cents) or a decimal library. At minimum, add epsilon-aware comparison.

#### BUG-303: Unauthenticated access when no API key configured
- **Crate**: `ironclad-server`
- **File**: `crates/ironclad-server/src/auth.rs`
- **Description**: When the server is launched without an API key in config (the default), the auth middleware skips all authentication. Any network-accessible client gets full admin access to all endpoints including config modification, session data, wallet operations, and agent control.
- **Impact**: Full unauthorized access on any network-exposed instance. Default config is insecure-by-default.
- **Fix**: Either (a) require an API key on first launch with a setup wizard, (b) bind to localhost-only by default, or (c) generate a random key on first run and display it.

#### BUG-304: SSRF via user-controlled provider URL
- **Crate**: `ironclad-server`
- **File**: `crates/ironclad-server/src/api/routes/admin.rs` (config update path)
- **Description**: The config update endpoint allows setting arbitrary `base_url` for LLM providers. A malicious or misconfigured client can point a provider to `http://169.254.169.254/latest/meta-data/` (AWS IMDS), `http://localhost:6379/` (Redis), or any internal service. All subsequent inference requests are proxied to the attacker-controlled URL with auth headers attached.
- **Impact**: SSRF allowing internal network scanning, cloud metadata theft, and credential exfiltration.
- **Fix**: Validate provider URLs against an allowlist or block private/link-local IP ranges. Strip auth headers when URL changes.

#### BUG-305: Pre-sanitization user message stored to DB (non-streaming)
- **Crate**: `ironclad-server`
- **File**: `crates/ironclad-server/src/api/routes/agent.rs` (message handler)
- **Description**: In the non-streaming message path, the raw user message is persisted to the database *before* injection detection and sanitization run. If injection detection subsequently rejects the message, the unsanitized payload remains in the DB and is visible via session history APIs.
- **Impact**: Stored XSS/injection payloads in session history; prompt injection content persisted even when blocked.
- **Fix**: Move DB persistence to *after* sanitization and injection checks pass.

#### BUG-306: Pre-sanitization user message stored to DB (streaming)
- **Crate**: `ironclad-server`
- **File**: `crates/ironclad-server/src/api/routes/agent.rs` (stream handler)
- **Description**: Same issue as BUG-305 but in the SSE streaming path. The message is persisted before the injection/sanitization pipeline.
- **Impact**: Same as BUG-305.
- **Fix**: Same as BUG-305.

#### BUG-307: A2A replay attack — no timestamp/nonce validation
- **Crate**: `ironclad-channels`
- **File**: `crates/ironclad-channels/src/a2a.rs`
- **Description**: The A2A (Agent-to-Agent) protocol verifies message signatures and encryption, but does not validate timestamps or nonces. A captured valid message can be replayed indefinitely. The `A2AEnvelope` has an optional `timestamp` field that is never checked.
- **Impact**: Replay attacks allow re-executing agent commands, re-triggering financial operations, or flooding with duplicate messages.
- **Fix**: Enforce mandatory timestamp within a configurable window (e.g., 5 minutes); add a nonce registry to reject duplicates.

#### BUG-308: Cron jobs never fire — naive datetime fails RFC3339 parse
- **Crate**: `ironclad-schedule`
- **File**: `crates/ironclad-schedule/src/cron.rs`
- **Description**: The cron worker computes `next_run_at` using `chrono::Local::now().naive_local().to_string()`, which produces `"2026-02-27 15:30:00"` format. The scheduler then compares this against stored RFC3339 timestamps (`"2026-02-27T15:30:00Z"`). The comparison always fails because the formats differ, so no cron job ever becomes "due".
- **Impact**: All cron-scheduled jobs silently never execute. Interval-based jobs may also be affected.
- **Fix**: Use `Utc::now().to_rfc3339()` consistently for all timestamp generation and comparison.

#### BUG-309: Ethereum recovery ID dropped — signatures unverifiable
- **Crate**: `ironclad-wallet`
- **File**: `crates/ironclad-wallet/src/ethereum.rs`
- **Description**: When signing Ethereum messages (EIP-191), the code extracts `(r, s)` from the ECDSA signature but discards the recovery ID `v`. The produced 64-byte signature cannot be verified by standard Ethereum tooling which expects 65 bytes (r + s + v). Any on-chain or off-chain signature verification fails.
- **Impact**: All Ethereum signatures produced by Ironclad wallets are unverifiable. Identity proofs, transaction signing, and message authentication are broken.
- **Fix**: Append the recovery ID byte (27 or 28 per EIP-191) to produce a 65-byte signature.

#### BUG-310: Wallet file permission race window
- **Crate**: `ironclad-wallet`
- **File**: `crates/ironclad-wallet/src/keystore.rs`
- **Description**: Wallet keystore files are created with default permissions, then `chmod 0600` is applied in a separate step. Between creation and chmod, the file is world-readable. On shared systems, another process can read the encrypted keystore during this window.
- **Impact**: Encrypted private key material briefly exposed on multi-user systems.
- **Fix**: Use `OpenOptions` with mode `0o600` on Unix (via `os::unix::fs::OpenOptionsExt`) to create the file with correct permissions atomically.

#### BUG-311: OpenAI streaming tool-call deltas silently dropped
- **Crate**: `ironclad-llm`
- **File**: `crates/ironclad-llm/src/providers/openai.rs`
- **Description**: When OpenAI streams a tool call across multiple SSE chunks, each chunk contains a delta with partial JSON arguments. The parser only processes complete chunks and drops deltas that span chunk boundaries. For complex tool calls (large JSON arguments), this silently truncates the tool call arguments.
- **Impact**: Tool calls with large arguments (e.g., code generation, structured data) produce truncated/invalid JSON, causing tool execution failures.
- **Fix**: Accumulate tool-call argument deltas in a buffer per tool-call index; only parse when the stream signals completion.

#### BUG-312: ContentGuard injection markers are case-sensitive
- **Crate**: `ironclad-agent`
- **File**: `crates/ironclad-agent/src/injection.rs`
- **Description**: The prompt injection detector checks for markers like `SYSTEM:`, `INSTRUCTION:`, `IGNORE PREVIOUS` using case-sensitive matching. Attackers trivially bypass with `System:`, `system:`, `SyStEm:`, or mixed-case variants. The homoglyph folding normalizes Unicode but does not normalize case.
- **Impact**: Prompt injection detection is trivially bypassable with case variation.
- **Fix**: Normalize to lowercase before pattern matching (after homoglyph folding).

#### BUG-313: Plugin manifest permissions unenforced
- **Crate**: `ironclad-plugin-sdk`
- **File**: `crates/ironclad-plugin-sdk/src/manifest.rs`
- **Description**: Plugin manifests declare `permissions` (filesystem, network, etc.) but the WASM sandbox and script executor never check these declarations against actual operations. A plugin declaring `permissions: []` can still access the network, read files, and execute arbitrary code.
- **Impact**: Plugin sandboxing is security theater — declared permissions are informational only.
- **Fix**: Enforce manifest permissions as WASM capability restrictions and script sandbox policies.

#### BUG-314: Script plugin timeout doesn't kill child process
- **Crate**: `ironclad-plugin-sdk`
- **File**: `crates/ironclad-plugin-sdk/src/script.rs`
- **Description**: When a script plugin exceeds its configured timeout, the runtime drops the future but does not send SIGKILL to the child process. The orphaned process continues running indefinitely, consuming resources.
- **Impact**: Malicious or hung plugins accumulate as zombie processes; resource exhaustion over time.
- **Fix**: On timeout, explicitly `child.kill()` and `child.wait()` before dropping.

#### BUG-315: Unbounded subprocess stdout/stderr reading
- **Crate**: `ironclad-plugin-sdk`
- **File**: `crates/ironclad-plugin-sdk/src/script.rs`
- **Description**: Script plugin stdout and stderr are read into memory without size limits. A malicious plugin can output gigabytes of data, causing OOM.
- **Impact**: Denial of service via memory exhaustion from a single plugin execution.
- **Fix**: Cap read buffer at a configurable maximum (e.g., 10MB) and truncate with a warning.

### HIGH Bugs (53)

#### BUG-316: Machine passphrase derived from guessable values
- **Crate**: `ironclad-core`
- **File**: `crates/ironclad-core/src/keystore.rs`
- **Description**: The "machine passphrase" for keystore encryption at rest is derived from `hostname + username + machine_id`. All three values are trivially discoverable. Any user on the same machine (or anyone who knows the hostname) can reconstruct the passphrase.
- **Impact**: Encrypted keystores are effectively protected by obscurity, not cryptography.

#### BUG-317: Bundled provider parse errors silently swallowed
- **Crate**: `ironclad-core`
- **File**: `crates/ironclad-core/src/provider.rs`
- **Description**: When parsing the bundled provider registry, individual provider parse failures are logged at `debug` level and skipped. A corrupt or incompatible registry entry silently disappears.
- **Impact**: Missing providers with no operator-visible indication.

#### BUG-318: Injection score boundary gap at 0.7 threshold
- **Crate**: `ironclad-agent`
- **File**: `crates/ironclad-agent/src/injection.rs`
- **Description**: Scores exactly equal to the threshold (0.7) are classified as "not injection" (`score < threshold`). The boundary should use `<=` or `>=` consistently. An attacker can craft payloads that score exactly 0.7 to evade detection.
- **Impact**: Marginal injection attempts at the boundary evade detection.

#### BUG-319: Compaction stores raw prompt text, not LLM summary
- **Crate**: `ironclad-core`
- **File**: `crates/ironclad-core/src/memory.rs`
- **Description**: Session compaction is supposed to summarize conversation history, but the implementation stores a static prompt template text ("Summarize the following conversation...") as the summary content, rather than actually calling the LLM to generate a summary.
- **Impact**: Compacted sessions lose all conversation context; retrieval returns the template text.

#### BUG-320: System prompt silently dropped when over context budget
- **Crate**: `ironclad-core`
- **File**: `crates/ironclad-core/src/context.rs`
- **Description**: Context assembly prioritizes user messages and retrieved memories. When the budget is tight, the system prompt is truncated or dropped entirely with no warning. The agent then operates without its core instructions.
- **Impact**: Agent behavior becomes unpredictable without system prompt; safety guardrails may be lost.

#### BUG-321: HMAC verification fails for trailing whitespace/newlines
- **Crate**: `ironclad-core`
- **File**: `crates/ironclad-core/src/trust.rs`
- **Description**: System prompt HMAC is computed over the raw prompt bytes. If the prompt file has trailing newlines (common in editors), but the runtime trims whitespace before verification, the HMAC check fails and the prompt is rejected as untrusted.
- **Impact**: Legitimate system prompts rejected after editing; operators disable HMAC checking as a workaround.

#### BUG-322: Speculative execution race condition
- **Crate**: `ironclad-agent`
- **File**: `crates/ironclad-agent/src/speculative.rs`
- **Description**: The speculative execution engine runs multiple candidate tool calls in parallel, then picks the best result. However, side-effecting tools (file writes, API calls, message sends) are not rolled back for losing candidates. All candidates' side effects persist.
- **Impact**: Duplicate side effects from speculative branches (double-sends, duplicate writes).

#### BUG-323: Spawn limit bypass during agent provisioning
- **Crate**: `ironclad-agent`
- **File**: `crates/ironclad-agent/src/spawn.rs`
- **Description**: The sub-agent spawn limit is checked before provisioning, but provisioning is async and the counter is incremented after spawn completes. Concurrent spawn requests during the provisioning window can exceed the limit.
- **Impact**: More sub-agents than configured limit, leading to resource exhaustion.

#### BUG-324: WASM memory limit only checks export table
- **Crate**: `ironclad-plugin-sdk`
- **File**: `crates/ironclad-plugin-sdk/src/wasm.rs`
- **Description**: The WASM sandbox enforces memory limits by checking the module's exported memory size, but WASM modules can use `memory.grow` at runtime to exceed the initial allocation. The runtime doesn't hook the `memory.grow` instruction.
- **Impact**: WASM plugins can allocate unlimited memory at runtime, bypassing sandbox limits.

#### BUG-325: Spoofable `x-user-id` header for actor identification
- **Crate**: `ironclad-server`
- **File**: `crates/ironclad-server/src/api/routes/mod.rs`
- **Description**: Actor identification for rate limiting and audit logging uses the `x-user-id` request header, which is client-controlled. Any client can spoof another user's identity for rate-limit evasion or audit log pollution.
- **Impact**: Rate limit bypass; audit log integrity compromised.

#### BUG-326: Global rate counter pre-increment race
- **Crate**: `ironclad-server`
- **File**: `crates/ironclad-server/src/rate_limit.rs`
- **Description**: The rate limiter increments the counter before checking the limit. Under burst traffic, all requests in a burst increment the counter, then all get rejected. The counter then reflects more "used" capacity than actual served requests, causing premature rejection in the next window.
- **Impact**: Rate limiter is overly aggressive under burst patterns; legitimate requests rejected.

#### BUG-327: DefaultHasher instability for actor fingerprinting
- **Crate**: `ironclad-server`
- **File**: `crates/ironclad-server/src/rate_limit.rs`
- **Description**: Actor fingerprints use `std::hash::DefaultHasher`, whose output is not stable across Rust versions or platforms. Rate limit state keyed by these hashes breaks after upgrades.
- **Impact**: Rate limit state invalidated on Rust toolchain updates; inconsistent across cluster nodes.

#### BUG-328: Telegram webhook secret header case mismatch
- **Crate**: `ironclad-channels`
- **File**: `crates/ironclad-channels/src/telegram.rs`
- **Description**: Telegram sends the webhook secret in `X-Telegram-Bot-Api-Secret-Token` header. The code checks for `x-telegram-bot-api-secret-token` (lowercase). While HTTP headers are case-insensitive per spec, some proxy configurations preserve original casing and axum's header map is case-sensitive by default.
- **Impact**: Webhook authentication silently fails behind certain reverse proxies.

#### BUG-329: WhatsApp verify_token uses non-constant-time comparison
- **Crate**: `ironclad-channels`
- **File**: `crates/ironclad-channels/src/whatsapp.rs`
- **Description**: The WhatsApp webhook verification endpoint compares `verify_token` using `==` (timing-sensitive). The API key auth uses `subtle::ConstantTimeEq`, but webhook tokens do not.
- **Impact**: Timing side-channel allows brute-forcing the WhatsApp verify token.

#### BUG-330: Config secret blocklist leakage via error messages
