# v0.8.0 Stabilization Plan

> Historical planning document. It describes pre-remediation state and should not be treated as current runtime behavior.

<!-- last_updated: 2026-02-27, status: DRAFT -->

This document defines the path from v0.8.0 feature freeze to release-ready,
covering four workstreams:

1. **Path to 95% test coverage** (unit, integration, regression)
2. **Architecture diagram remediation**
3. **Bug discovery program** (feature-freeze exit criteria)
4. **Iterative fix-and-retest cycle**
5. **Smoke & UAT testing** (patched codebase + running instance)

## 2. Architecture Diagram Remediation

### Current State

**60 diagram issues found and fixed** (BUG-001 through BUG-058):
- 33 C4 model drift issues (all Fixed)
- 14 dataflow diagram drift issues (all Fixed)
- 12 sequence diagram drift issues (all Fixed)
- 1 circuit-breaker audit drift (Fixed)

### Remaining Risk: Continuous Drift

The codebase has 40+ Mermaid diagrams across 20+ files. As bug fixes land during
stabilization, diagrams can drift again. The following process prevents that.

### Diagram Validation Protocol

#### Step 1: Post-Fix Diagram Audit

Every PR that modifies public API surface, adds/removes modules, or changes
inter-crate wiring must include:

1. **Automated check:** CI step validates that `docs/architecture/` files were touched
   if `crates/*/src/lib.rs` or `Cargo.toml` files changed (advisory warning, not blocking).

2. **Manual audit checklist** (PR template):
   ```
   - [ ] C4 component diagram updated if modules added/removed
   - [ ] Sequence diagram updated if cross-crate flow changed
   - [ ] Dataflow diagram updated if data path changed
   - [ ] Cross-reference matrix in ironclad-design.md updated
   ```

#### Step 2: Structural Validation Tests

Add a test in `ironclad-tests` that validates diagram-to-code consistency for
the highest-drift areas:

| Validation | Method |
|-----------|--------|
| Module existence | Parse `lib.rs` `pub mod` list; compare to C4 component names |
| Dependency edges | Parse `Cargo.toml` deps; compare to C4 container `Rel()` lines |
| Route count | Count `.route()` calls in `routes/mod.rs`; compare to API.md |
| Table count | Count `CREATE TABLE` in `schema.rs`; compare to design doc |
| Heartbeat tasks | Count `HeartbeatTask` enum variants; compare to schedule C4 |

These tests belong in `ironclad-tests/src/doc_consistency.rs` and run as part of
`just test-release-critical`.

#### Step 3: Known Remaining Issues

| Area | Issue | Action |
|------|-------|--------|
| Response Transform Pipeline | `transform.rs` is dead code (BUG-024/039) | Either delete file or wire it into `pub mod` |
| TLS Sequence Diagram | Describes unimplemented TLS (BUG-054) | Mark as "Design Target" or remove |
| Context Observatory | References non-existent structs (BUG-052) | Mark as "Design Target" or remove |
| `agent_turn_legacy` | 3 diagrams reference deprecated noop (BUG-030/036/050) | Verify removal in code, then remove from diagrams |

## 4. Iterative Fix-and-Retest Cycle

### Process Per Bug Fix

```
┌─────────────────────────────────────────────────────────┐
│                    FIX CYCLE                            │
│                                                         │
│  1. Pick highest-priority open bug from Tier 1          │
│  2. Create branch: fix/BUG-XXX-short-description        │
│  3. Write failing regression test FIRST                 │
│  4. Implement fix (minimal, focused)                    │
│  5. Verify fix: regression test passes                  │
│  6. Run: just test-crate <affected>                     │
│  7. Run: just test-regression                           │
│  8. Run: just coverage-crate <affected>                 │
│  9. Update bug-ledger.md: status → Fixed                │
│  10. Update regression-matrix.md: add new ID            │
│  11. PR with checklist (diagram audit if applicable)    │
│  12. Merge → CI validates all gates                     │
│  13. After merge: just coverage-update-baseline         │
│                                                         │
│  Repeat until Tier 1 empty, then Tier 2, then Tier 3   │
└─────────────────────────────────────────────────────────┘
```

### Retest Cadence

| Gate | When | Command |
|------|------|---------|
| Unit (affected crate) | Every fix PR | `just test-crate <crate>` |
| Regression battery | Every fix PR | `just test-regression` |
| Integration suite | Every 3 fixes | `just test-integration` |
| Full workspace | Every 5 fixes | `just test-release-critical` |
| Soak/fuzz | Weekly | `just test-soak-fuzz` |
| Coverage gate | Every fix PR | `just coverage-check` |
| Mutation testing | Per crate touched | `just mutants <crate>` |

### Fix Batch Strategy

Group related bugs for efficiency:

| Batch | Bugs | Theme | Affected Crates |
|-------|------|-------|-----------------|
| B1 | BUG-061, BUG-062 | Telegram parity | channels, agent, server |
| B2 | BUG-064, BUG-070 | OAuth security | llm, core |
| B3 | BUG-063 | Windows daemon | server |
| B4 | BUG-069, BUG-074, BUG-088 | Zero-coverage critical | db, wallet, plugin-sdk |
| B5 | BUG-065, BUG-066 | Supply chain + permissions | plugin-sdk, agent |
| B6 | BUG-071..BUG-077 | Coverage ramp (browser, schedule, server) | browser, schedule, server |
| B7 | BUG-079..BUG-085 | Coverage ramp (agent, db, server) | agent, db, server |
| B8 | BUG-086..BUG-094 | Mutation kill rate | core, plugin-sdk, schedule |

### Verification Checkpoints

After each batch, run full verification:

```bash
# After batch completion
just test-release-critical          # workspace + integration + regression
just coverage-check                 # 80% floor (ratcheting toward 95%)
just mutants <affected-crate>       # <15% survival target
```

After all Tier 1 bugs fixed:
```bash
just test-v080-go-live              # Full release gate
```

## 5. Smoke & UAT Testing

### Scope

Smoke and UAT tests validate the **running system** (not just compiled tests).
They verify operator-critical workflows end-to-end against a live server.

### 5A. Patched Codebase Smoke Testing

After each fix batch, build and run the server from source:

```bash
# Build patched binary
cargo build --release

# Start server with test config
cargo run --release --bin ironclad -- serve -c test-config.toml &
SERVER_PID=$!

# Wait for server ready
sleep 3

# Run CLI smoke
just test-uat-cli

# Run Web/API smoke
just test-uat-web

# Run full smoke suite
just smoke

# Cleanup
kill $SERVER_PID
```

### 5C. Channel-Specific Smoke Tests

Given BUG-061/062 (Telegram parity violations), add per-channel smoke:

| Channel | Test Method | Validates |

| HTTP API | `POST /api/agent/message` | Response parity with Telegram |
| Discord | Send via bot token | Response quality (if configured) |

Record response text and compare semantic similarity (manual review or LLM-as-judge).
